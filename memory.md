## 2026-07-23 — nightly refresh: two overlapping failures untangled
- Symptom: `refresh-index.yml` "failed again". Turned out to be **two unrelated causes** masking each other:
  1. **Persistent (07-14 → 07-22):** every nightly rebuilt the full EN+JA index (30–60 min) then died at the final step — `pull request create failed: GraphQL: GitHub Actions is not permitted to create or approve pull requests`. The `GITHUB_TOKEN` couldn't open the refresh PR into protected `main`, so the rebuilt index was discarded nightly. Fix was a repo setting, not code: Settings → Actions → General → **"Allow GitHub Actions to create and approve pull requests"** (API: `can_approve_pull_request_reviews` on `/actions/permissions/workflow`; authoritative here because `TheSaltyKorean` is a User account, no org override). Flag got flipped on ~midday 07-22 (PR #16 merged 17:48 that day proves it worked from then).
  2. **Transient (07-23):** `api.pokemontcg.io` returned `500` on the *initial* `/v2/sets` call. Unlike a per-set `500` (which build-index.mjs skips and continues), a sets-*list* failure is fatal — killed the run at 16s. `fetchJson` retried only 3× over ~5s, too short to outlast the blip.
- Verified the permission fix **end-to-end** by manual `workflow_dispatch` (run 30003883977, 32m): it created+merged **PR #18** (`bfc4ef2` on main) — today's refresh published, backlog cleared.
- Hardened `fetchJson` (**PR #17**): 5 attempts, exponential backoff (2/4/8/16s, ~30s patience), honor `Retry-After`, retry network/429/5xx but **fail fast on 4xx** (a 404 was previously retried like a transient error). Network faults now caught only around `fetch()` so an HTTP-status throw doesn't loop the retry path. Six mocked-fetch cases pass.
- LESSON: when a scheduled job "keeps failing", check whether consecutive failures share a cause — here a long-standing PR-permission failure and a one-off upstream 500 landed the same morning and looked like one unfixed bug. Compare failure *durations* (16s vs 30–60m) and the failing *step* to tell them apart.
